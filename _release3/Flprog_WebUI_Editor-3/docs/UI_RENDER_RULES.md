# Правила отрисовки UI (без редактирования)

Документ задаёт правила, достаточные для воспроизведения того же вида интерфейса при отрисовке без логики редактирования (превью, экспорт, рантайм).

---

## 1. Канва

- **Размеры** в логических единицах: `width`, `height`. Цвет фона: `color` (HEX).
- **Масштаб отображения**: один общий `scale` для всего содержимого. Размер на экране: `displayWidth = width * scale`, `displayHeight = height * scale`.
- Вся графика рисуется в **логических координатах** (0…width, 0…height), затем применяется общая трансформация масштабирования `scaleX=scale`, `scaleY=scale`. Масштабирование выполняется на ширину экрана

---

## 2. Сетка (опционально)

- не нужна

---

## 3. Система координат виджетов

- У каждого виджета: `x`, `y` — позиция левого верхнего угла в логических единицах канвы; `width`, `height` — размеры в тех же единицах.
- При отрисовке: группа виджета помещается в точку `(x, y)`; внутреннее содержимое рисуется в локальных координатах с шириной `lw = width`, высотой `lh = height`.

---

## 4. Текст: общие метрики

Для текста внутри виджета (кнопка, поле ввода, Label, значение слайдера):

- **Высота области текста**: `textHeight = 0.85 * lh`.
- **Смещение по вертикали**: `textY = (lh - textHeight) / 2`.
- **Размер шрифта**: `textFontSize = max(8, textHeight * 0.65)`.

Подпись над виджетом (caption) не использует эти формулы (см. п. 6).

---

## 5. Цвет текста

- **Подпись над виджетом (caption)**: цвет текста = `contrastColor(цвет_канвы)`.
- **Кнопка, поле ввода, значение слайдера**: цвет текста = `contrastColor(цвет_виджета)`. Для переключателя (switch) «цвет виджета» — это цвет текущего состояния (#22c55e или #e5e7eb).
- **Label (метка)**: цвет текста **строго** берётся из данных виджета: `fill = color` виджета. Не пересчитывается по контрасту и не зависит от фона канвы.

Функции: `contrastColor(hexOrRgb)`, `getLuminance(hexOrRgb)` — в `contrastColor.ts`.

---

## 6. Отрисовка по типам виджетов

### Подпись (caption)

- Если тип не `rect` и `caption` не пустой: текст рисуется **над** виджетом.
- Позиция: `x=0`, `y=-18`, `width=lw`, `height=18`, выравнивание по центру по горизонтали и вертикали.
- Цвет: `contrastColor(цвет_канвы)`, `fontSize=12`.

### LED

- Один круг: `radius = lw/2`, центр `(lw/2, lh/2)`.
- Заливка: при `text === '1'` — `color` виджета, иначе `#333`.
- Обводка: `#111`, `strokeWidth=2`.
- Тень: при включённом состоянии `shadowBlur=15`, `shadowColor=color` виджета; иначе без тени.

### Label

- Один текстовый блок: содержимое `text`, область `(0, textY, lw, textHeight)`.
- Выравнивание: по горизонтали — left, по вертикали — middle.
- Цвет текста: **только** `color` виджета (см. п. 5).
- Шрифт: `textFontSize` по п. 4.

### Button, Input, Switch, Rect (прямоугольник)

- **Прямоугольник** (Rect):
  - Размер: `lw × lh`.
  - Заливка: для switch — при `text === '1'` цвет `#22c55e`, иначе `#e5e7eb`; для остальных — `color` виджета.
  - Обводка: только у input — `#ddd`, `strokeWidth=1`; у остальных без обводки.
  - Скругление: button — 5, input — 2, switch — `lh/2`, rect — 10.

- **Switch дополнительно**: круглый «ползунок» — круг радиусом `lh/2 - 3`, центр по горизонтали при `text === '1'` в `lw - lh/2`, иначе в `lh/2`; по вертикали `lh/2`. Заливка `#ffffff`, лёгкая тень.

- **Текст на кнопке и в поле ввода**: один текстовый блок в области `(0, textY, lw, textHeight)`, выравнивание center/middle, цвет `contrastColor(цвет_виджета)`, размер шрифта по п. 4.

### Slider

- Прямоугольник как у button/input (без отдельной «дорожки»), плюс текст **справа** от виджета: позиция по x = `lw + 8`, область по ширине 50, по высоте `textHeight`; выравнивание left/middle; содержимое — значение `text`; цвет `contrastColor(цвет_виджета)`, размер шрифта по п. 4.

---

## 7. Что не рисуется при «чистой» отрисовке

- Нет выделения (Transformer), нет осей центра канвы, нет контура по краям виджета.
- Нет элементов интерфейса редактирования
- Нет сетки

---

## 8. Константы

- `GRID_SIZE = 5` — для сетки и (при необходимости) привязки координат.
- `contrastColor`, `getLuminance` — из `src/contrastColor.ts`.

Этих правил достаточно для воспроизведения внешнего вида UI при отрисовке без редактирования.

## 9. Режим отрисовки без редактора с данными от МК (рантайм)

Описанный ниже режим предназначен для клиента (браузер, приложение), который **не редактирует** интерфейс, а только запрашивает конфигурацию и состояние у МК и рисует UI по правилам из п. 1–8.

### 9.1. Источники данных

- **Конфигурация интерфейса**: GET `/config`. Ответ — JSON без обёртки: массив `[tabIds, compactWidgets]`.
- **Текущее состояние переменных**: GET `/state`. Ответ — JSON-объект: ключи — имена переменных (см. ниже), значения — текущие значения с МК.

### 9.2. Формат конфигурации (`/config`)

- Ответ — массив из 2 или 3 элементов: `[tabIds, compactWidgets]` или `[tabIds, compactWidgets, canvas]`.
- `tabIds` — массив строк, идентификаторы вкладок (порядок важен).
- `compactWidgets` — массив виджетов. Каждый виджет — массив:
  - `[0]` — `id` (string).
  - `[1]` — `typeCode` (number): 0 = button, 1 = slider, 2 = input, 3 = led, 4 = label, 5 = rect, 6 = switch.
  - `[2]` — `x` (number).
  - `[3]` — `y` (number).
  - `[4]` — `width` (number).
  - `[5]` — `height` (number).
  - `[6]` — `color` (string, HEX).
  - `[7]` — `text` (string), начальный/подставляемый текст.
  - `[8]` — `tabIndex` (number), индекс вкладки в `tabIds`.
  - `[9]` — опционально: `varName` (string), если переменная привязана и имя отличается от `id`.

- `canvas` (опционально, 3-й элемент) — объект `{ width, height, color }` в логических единицах и HEX цвета фона. Если передан, клиент использует эти размеры канвы вместо вычисления по границам виджетов.

При отрисовке рантайм-клиент должен восстановить для каждого виджета: `id`, `type` по `typeCode`, `x`, `y`, `width`, `height`, `color`, `text`, `tabId = tabIds[tabIndex]`, при наличии — `varName`. Поле `caption` в компактном формате не передаётся; при необходимости его можно считать пустым или не рисовать подпись.

### 9.3. Формат состояния (`/state`)

Поддерживаются два формата (выбор по параметру запроса `fmt`).

**По умолчанию (GET `/state`)** — массив по индексам (вариант 10.1):

- Ответ — JSON-массив значений в фиксированном порядке: сначала по одному значению на каждый виджет с переменной (в том же порядке, что в конфиге `compactWidgets` — только виджеты с `varType !== 'none'`), затем `sound_enabled` (если такой переменной нет среди виджетов), затем `ui_message`.
- Пример: `[1,0,50,true,""]` — индексы 0,1,2 — переменные виджетов, 3 — sound_enabled, 4 — ui_message.
- Клиент сопоставляет индекс с виджетом по порядку виджетов с переменными в конфиге; последние два элемента — всегда `sound_enabled` и `ui_message` (если конфиг не содержит виджет с `sound_enabled`, иначе — один элемент `ui_message` в конце).

**Короткие ключи (GET `/state?fmt=short`)** — вариант 10.2:

- Ответ — JSON-объект с короткими ключами: `"0"`, `"1"`, `"2"`, … (по одному на каждую переменную виджета в том же порядке), затем `"s"` — sound_enabled, `"m"` — ui_message.
- Пример: `{"0":1,"1":0,"2":50,"s":true,"m":""}`.
- Порядок ключей 0,1,2,… совпадает с порядком виджетов с переменными в конфиге; клиент строит соответствие по конфигу.

**Устаревший формат (полные имена переменных)** в новой генерации не используется; рантайм-клиент должен поддерживать форматы массива и/или коротких ключей.

Ниже — описание типов значений (как в объекте с полными именами):

- Для виджетов **только на чтение** (LED, Label): ключ = `varName`; значение — число (int/float), строка или boolean в зависимости от типа переменной.
- Для виджетов **двунаправленных** (switch, slider, input): ключ = `varName_in` (на МК в коде это переменная «вход» после пользовательского ввода). Значение — число, строка или boolean.
- Дополнительно могут быть: `sound_enabled` (boolean), `ui_message` (string).

Типы значений в JSON: числа как числа, boolean как `true`/`false`, строки в кавычках.

### 9.4. Сопоставление состояния с виджетами

- **Формат «массив» (GET `/state`):** из конфига построить список виджетов с переменными в том же порядке, что в `compactWidgets` (только элементы с переменной). Индекс `i` в этом списке соответствует `state[i]`. Элементы `state[n-2]` и `state[n-1]` (или один элемент в конце при наличии виджета sound_enabled) — это `sound_enabled` и `ui_message` (см. п. 9.3).
- **Формат «короткие ключи» (GET `/state?fmt=short`):** ключи `"0"`, `"1"`, … соответствуют тому же порядку виджетов с переменными; `"s"` — sound_enabled, `"m"` — ui_message.
- Если виджет без переменной (`varType === 'none'` или аналог), значение для отображения брать только из конфига (`text`), состояние не применять.
- Имя переменной для виджета: из элемента конфига `[9]` если есть, иначе `id`.

### 9.5. Значение для отрисовки (display value)

По типу виджета и значению из state (или конфига) вычислить одно «отображаемое» значение, которое в правилах п. 4–6 фигурирует как `text` виджета (или аналог):

- **LED**: отображаемое значение — строка `"1"` или `"0"`. Из state: boolean → `true` → `"1"`, иначе `"0"`; число → ненулевое → `"1"`, иначе `"0"`.
- **Switch**: так же `"1"` / `"0"` для заливки и положения ползунка.
- **Button**: для визуального «нажатия» можно использовать значение из state аналогично (например `"1"` затемнённая); иначе показывать только подпись/текст из конфига.
- **Slider**: отображаемое значение — число 0…100. Из state взять число (int/float), ограничить: `Math.min(100, Math.max(0, value))`; для отрисовки ползунка и текста справа использовать это число (можно округлить).
- **Input**: отображаемое значение — строка из state (ключ `varName_in`).
- **Label**: отображаемое значение — строка или приведённое к строке значение из state (ключ `varName`); если переменной нет — текст из конфига.
- **Rect**: состояние не используется; отрисовка только по конфигу (прямоугольник, цвет).

После вычисления подставить это значение в поле `text` (или в отдельное поле display value) виджета и применять правила отрисовки из п. 4–6 без изменений.

### 9.6. Вкладки (tabs)

- Конфиг задаёт `tabIds` и у каждого виджета — `tabIndex`. Рантайм-клиент может рисовать только виджеты текущей вкладки: отфильтровать по `tabId === tabIds[activeTabIndex]`.
- Переключение вкладок — только смена активного индекса; запрос конфига один раз при загрузке.

### 9.7. Итог для рантайм-клиента

1. Загрузить конфиг: GET `/config` → разобрать `[tabIds, compactWidgets]`, восстановить список виджетов с полями `id`, `type`, `x`, `y`, `width`, `height`, `color`, `text`, `tabId`, `varName` (если есть).
2. Периодически или по событию запрашивать состояние: GET `/state` → объект `state`.
3. Для каждой вкладки (или только активной) для каждого виджета: по п. 9.4 определить ключ в `state`, по п. 9.5 вычислить display value.
4. Отрисовать канву и виджеты по правилам п. 1–8, используя для отображения вычисленные display value вместо статического `text` там, где это задано п. 9.5.
5. Редактирование (перетаскивание, ресайз, панели) не используется; запросы на изменение переменных (POST/GET `/set?var=...&value=...`) — отдельно, по логике управления с МК.

## 10. Варианты оптимизации протокола общения с МК

Ниже — возможные улучшения без изменения базовой семантики (конфиг + состояние + set). Реализация зависит от поддержки на стороне Flprog WebServer и от выбора формата на клиенте.

### 10.1. Состояние по индексам (минимальный объём)

- Вместо JSON-объекта с именами переменных отдавать **массив значений в фиксированном порядке**, совпадающем с порядком виджетов с переменными в конфиге.
- Пример: вместо `{"led1":1,"switch1_in":0,"slider1_in":50}` ответ: `[1,0,50]`.
- Клиент знает порядок из `/config` и сопоставляет по индексу. Экономия: нет повторения имён переменных при каждом запросе `/state`, меньше RAM на МК (один буфер под массив).

### 10.2. Короткие ключи в state

- Если сохранять формат «ключ–значение», использовать короткие ключи (1–2 символа или числа), заданные в конфиге или жёстко: например `"0":1,"1":0,"2":50`. Клиент строит таблицу «индекс → varName» из конфига.

### 10.3. Один запрос «конфиг + состояние»

- Эндпоинт вида GET `/init` или GET `/` с параметром `?full=1`: в одном ответе отдать и конфиг, и текущее состояние (например, два JSON-объекта/массива подряд или один составной). Убирает один round-trip при первой загрузке.

### 10.4. Пакетная установка переменных (batch set)

- Один запрос на изменение нескольких переменных: например POST `/set` с телом `{"var1":val1,"var2":val2}` или форматом `var=name1&value=v1&var=name2&value=v2`. Меньше запросов при массовом обновлении с клиента.

### 10.5. Условная выдача конфига (ETag / версия)

- Конфиг меняется редко. Добавить заголовок `ETag` или поле `configVersion` в ответ `/config`; клиент при следующих загрузках отправляет `If-None-Match` или `version=...`. МК при неизменном конфиге возвращает 304 Not Modified без тела. Экономия трафика и времени.

### 10.6. Keep-Alive вместо Connection: close

- Если WebServer поддерживает, держать соединение открытым для серии запросов (конфиг, затем периодические state/set). Снижает накладные расходы на повторные TCP/HTTP handshake при частом опросе.

### 10.7. Редкий опрос / только при изменениях

- На клиенте: увеличить интервал опроса `/state` где допустимо по логике UI.
- На МК (если библиотека позволит): отдавать состояние по запросу без накопления большого буфера; при возможности — push (SSE/WebSocket) только при изменении переменных, чтобы не опрашивать по таймеру.

### 10.8. Сжатие (gzip)

- Включить сжатие ответов для `/config` и `/state`, если WebServer на МК это поддерживает. На малом объёме данных выигрыш может быть небольшим; имеет смысл при большом конфиге или частом опросе.

### 10.9. Приоритет внедрения

- Быстро и с максимальной отдачей: **10.1** (state как массив по индексам) и **10.4** (batch set).
- Далее: **10.3** (один запрос конфиг+состояние), **10.5** (ETag для конфига), **10.6** (Keep-Alive).
- Остальное — по возможностям стека и требованиям к задержкам/трафику.
